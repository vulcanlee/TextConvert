# 文件轉純文字設計

- 文件用途:說明 TextConvert 的整體架構、資料流與關鍵設計決策。
- 主要讀者:開發者、維護人員。
- 文件版本:1.0
- 對應系統版本:1.0.0
- 建立日期:2026-07-15
- 編碼:UTF-8 繁體中文

## 一、問題背景

需將多種文件格式的內容擷取為統一的純文字,作為知識庫素材;需可批次處理、
可設定、有完整日誌,且單一檔案失敗不影響整批作業。

## 二、需求決策

| 決策點 | 選擇 | 理由 |
|--------|------|------|
| Office 格式 | 僅新版 OOXML | 使用免費 OpenXML SDK,不引入舊版二進位相依 |
| 目錄掃描 | 遞迴 | 完整涵蓋子目錄素材 |
| 輸出結構 | 鏡像子目錄 | 對應來源、避免同名衝突 |
| 輸出檔名 | 保留原副檔名(`x.ext.txt`) | 徹底避免同名衝突並保留來源格式線索 |
| PDF | 僅抽內嵌文字,不 OCR | 降低相依與複雜度 |
| 日誌 | Serilog | 由設定檔控制等級與 Console / 檔案輸出 |
| 相依注入 | 手動組合,不用 DI 容器 | 需求單純,避免過度設計(YAGNI) |

## 三、整體架構

```
Program.cs(組合根)
  ├─ 載入 appsettings.json(Microsoft.Extensions.Configuration)
  ├─ 初始化 Serilog(Console + 每日滾動檔案)
  ├─ CommandLineParser 覆寫來源 / 輸出目錄
  └─ 組合 ConverterRegistry 與 ConversionService 後執行

ConversionService(協調)
  ├─ FileScanner        逐檔列舉(遞迴可選)
  ├─ ConverterRegistry  依副檔名解析轉換器
  └─ 逐檔:轉換 → 鏡像寫出 → try/catch 記錄 → 統計

Converters(各格式)
  IDocumentConverter ← Word / Excel / PowerPoint / Pdf / Html / Markdown / Csv / Text
```

## 四、資料流

1. 輸入:來源目錄、輸出目錄、遞迴 / 覆寫設定。
2. `FileScanner.EnumerateFiles` 產生檔案路徑序列。
3. 每個路徑經 `ConverterRegistry.Resolve` 取得對應轉換器(無則跳過)。
4. 轉換器 `Convert(path)` 回傳純文字字串。
5. `ConversionService` 計算鏡像輸出路徑 `輸出目錄 + 相對子路徑 + 原檔名 + ".txt"`,
   以 UTF-8(無 BOM)覆寫寫出。
6. 輸出:鏡像結構的 `*.txt` 檔與 `logs/` 日誌;回傳各檔 `ConversionResult`。

## 五、關鍵設計要點

- **`IDocumentConverter` 介面**:`CanHandle(extension)` 與 `Convert(filePath)`,
  每個格式一個單一職責實作,易於獨立測試與日後擴充。
- **容錯**:單檔轉換以 try/catch 包裹,失敗記 Error 後續處理,不中斷整批。
- **輸出編碼**:統一 UTF-8(無 BOM),確保中文內容無亂碼。
- **可擴充點**:新增格式僅需實作 `IDocumentConverter` 並在 `Program.cs` 註冊。
